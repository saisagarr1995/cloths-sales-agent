# WhatsApp Cloth-Sales Agent — Implementation Plan

## Context

Build a WhatsApp sales agent for a clothing business (kids, girls, women — boys/men later) that handles enquiries, shows the catalog (with/without images), recommends best-sellers with reasons, suggests occasion-based items, remembers customer preferences, handles discount objections in English/Tamil/other Indian languages, collects the delivery address, sends a clean order summary with a UPI payment link + QR, and confirms the order once the customer reports payment. The project must be lightweight, open-source-only, token-efficient, fully tested with auto-bug-fixing, and carry cross-session context files so any new Claude session can resume exactly where the last one stopped.

## Confirmed decisions (user-approved)

| Decision | Choice |
|---|---|
| WhatsApp connection | **Baileys** (`@whiskeysockets/baileys`) — open-source, QR-scan login on a normal WhatsApp number. Known risk (accepted): unofficial, number can be banned for spam-like behavior. |
| AI brain | **Groq free tier** (Llama 3.3 70B via Groq's free API) behind a small provider wrapper so the LLM can be swapped with a one-line config change. No Claude API. |
| Payments | **UPI deep link + QR** (`upi://pay?...`) with manual confirmation — customer pays, sends "paid"/UTR/screenshot, owner gets a WhatsApp alert to verify. No gateway, no fees. |
| Catalog | **Sample data** — ~18 realistic products (frocks, kurtis, sarees, leggings, etc.) in editable `data/catalog.json`; placeholder images the user replaces later. |
| Runtime | **Node.js** (single runtime; Baileys is Node). No Python, so no venv needed. |

## Architecture — token-efficient hybrid brain

1. **Deterministic layer first (zero LLM tokens):** greetings/any first message → instant welcome; "menu"/"catalog"/"price list" → formatted text menu from catalog.json; "with images"/"photos" → catalog images; "show me [product]" → that product's image+details; "paid"/UTR → payment-claim handling. Multilingual keyword matching (English + Tamil + Hindi transliterations).
2. **LLM layer (Groq) only for free-form conversation:** objection handling, occasion recommendations, best-seller pitches, preference memory, any Indian language. The LLM receives a compact system prompt (condensed catalog + customer memory + conversation state + selling playbook) and returns JSON: `{reply, actions[]}` where actions are things like `add_to_cart`, `send_product_image`, `set_preference`, `start_checkout`. History windowed to last ~10 messages to cap tokens.
3. **Order state machine per customer:** `browsing → cart → collecting_address → awaiting_payment → payment_claimed → confirmed`. Deterministic code owns checkout (address collection, order summary, UPI link) — the LLM never computes prices.
4. **Persistence:** plain JSON files — `data/customers/<phone>.json` (memory: preferences, language, cart, state) and `data/orders.json`. No database. Survives restarts.

## Dependencies (5 packages only, all open-source)

`@whiskeysockets/baileys` (WhatsApp), `qrcode` (UPI QR PNG), `qrcode-terminal` (login QR display), `pino` (logger Baileys needs, set silent), `dotenv` (.env config). Groq is called with Node's built-in `fetch` — no SDK needed. Placeholder product images generated by a zero-dependency script using Node's built-in `zlib` (solid-color PNGs; captions carry product name/price).

## Cross-session context tracking (rule 3)

- `CLAUDE.md` is auto-loaded by Claude Code every session. It will instruct: read `PLAN.md` (the frozen plan) and `PROGRESS.md` (current status + next step) before doing anything; never deviate from the plan without user approval; update `PROGRESS.md` and `RECREATE_PROMPT.md` after every change; token-economy rules (no experimenting, no extra packages).
- `PROGRESS.md` ends with a "NEXT STEP" line so a fresh session knows exactly where to continue.

## Skills in `.claude/skills/` (rule 2)

1. **update-docs** — after any code/plan change, update `PROGRESS.md` (what was done, next step) and `RECREATE_PROMPT.md` in sync.
2. **test-and-fix** — run the full test suite; on any failure, diagnose, fix, re-run until green; then log results in PROGRESS.md. (rule 5: automatic bug fixing)
3. **add-product** — safely add/edit/remove a product in `catalog.json` and regenerate its placeholder image.

## Selling behavior (system prompt for the LLM)

Persuasive but honest boutique salesperson; mirrors the customer's language (English/Tamil/any Indian language); recommends exactly 2 best-sellers with the "why" from catalog data; occasion-aware suggestions; discount objections → value framing + small nudge back to order (never invents discounts, may mention free-delivery-over-₹999 style policy from config); remembers stated preferences via `set_preference` action; always steers toward closing the order.

## Payment flow

Checkout: collect address (deterministic prompts) → order summary (items, sizes, qty, total, address) → `upi://pay?pa=<UPI_ID>&pn=<name>&am=<total>&tn=ORD<id>` link **and** a QR PNG → customer replies "paid"/UTR/screenshot → order marked `payment_claimed`, customer gets confirmation message, owner's WhatsApp number gets the full order + claim for manual verification → owner replies "confirm ORD<id>" from their number → order `confirmed`, customer notified.

## Build order

1. Scaffold: `package.json`, `.env.example`, `config.js`, docs (`CLAUDE.md`, `PLAN.md`, `PROGRESS.md`, `RECREATE_PROMPT.md`), skills.
2. `data/catalog.json` + `scripts/gen-images.js` + generated images.
3. `catalog.js`, `store.js`, `payment.js` (pure logic, no WhatsApp yet).
4. `intents.js` + `handler.js` (deterministic layer + state machine) against the fake transport.
5. `brain.js` (Groq wrapper + actions).
6. `index.js` (Baileys wiring).
7. Tests: deterministic suite, then LLM suite; run test-and-fix loop until green.
8. Final doc sync; live-test instructions for the user (scan QR, message from another phone).

## Verification

- `npm test` — deterministic suite runs with no network; LLM suite runs when `GROQ_API_KEY` is set (free). All scenarios from the requirements are covered: instant welcome, menu, menu-with-images, single product image, 2 best-sellers with reasons, occasion suggestions, preference memory, Tamil/English discount objection → back to order, full checkout → address → summary → UPI link+QR → paid → owner alert → confirm.
- Live check: user fills `.env` (Groq key from console.groq.com, their UPI ID, owner number), runs `npm start`, scans QR with the business WhatsApp, sends "Hi" from another phone.
